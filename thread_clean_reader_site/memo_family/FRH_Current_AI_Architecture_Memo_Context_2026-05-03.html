FRH Current AI Architecture Memo Context

Date: 2026-05-03

Why These Updated Memos Were Written

On 2026-03-24, a first architecture memo set was written to explain the then-current FRH AI operating model. Those memos accurately described a role-specialized local workflow centered on CTC, GMA, Parser, and DR_Lab. Since then, the project has advanced far enough that an updated set is warranted.

The most important change is not that the earlier architecture became obsolete. It is that the architecture proved durable, portable, and extensible. The same core pattern still applies, but it now includes a stronger publication lane, a more explicit local-agent operating model, and a much harder-won understanding of recovery and vendor risk.

What Changed Since March

Several practical developments forced a broader and more current description:

  • Web_Reader became a real downstream publication lane rather than a hypothetical future surface.
  • The active CTC lane matured into a Codex-centered local agent workspace operating directly on files and tools rather than behaving like a conventional browser chat alone.
  • The architecture proved portable across model and tool environments, including Codex, Claude Code, Gemini, and even limited local use of Gemma 4 E4B with an 8K context window on a 3080.
  • The project gained more lived evidence around publication, staging, handoff memos, source maps, batch logs, and final web-facing promotion.
  • A much sharper recovery lesson emerged: full local export of agent chat transcripts is mandatory if the system is to survive token pricing shocks, vendor account closures, or unexplained platform lockouts.

What The Architecture Has Now Produced

The updated memos also need to reflect that the architecture is no longer justified only by intermediate workflow success. It has now produced concrete, high-quality, public-facing outputs in a domain that is normally very hostile to structured computing support.

Two especially important examples are:

  • the Sample Character Library, where CTC coordinated character engineering, images, master HTML assembly, PDF refresh cycles, verification, staging, and handoff into Web_Reader
  • the New Player Packet website, where GMA-adjacent reasoning and CTC authoring work helped transform explanatory material into a player-facing web-ready structure that Web_Reader could polish and publish

Both of these local efforts were then packaged, promoted, and sent through Web_Reader for polishing and publication to GitHub Pages. That is exactly the kind of evidence these memos now need to emphasize: this architecture is producing real-world outputs, not merely internal experiments.

The New Discovery That Matters Most

The single most important discovery since the March memos is that the full exported chat history of the agent lanes must be preserved locally as part of the architecture itself.

This is not mere convenience logging. It is structural recovery protection. The reasons are now practical rather than theoretical:

  • vendor token pricing can change suddenly
  • accounts can be closed or restricted with no warning
  • platform behavior can change faster than a working project can be refactored
  • cross-model portability depends on having the operational memory in local files rather than only inside a vendor-hosted conversation history

In other words, transcript export has become part of disaster recovery, not just part of note-taking.

What These New Memos Need To Capture

The updated memo set therefore needs to explain:

  • that the old role-separated architecture still stands
  • that Web_Reader is now a first-class lane
  • that local-agent tooling made the system stronger, but did not replace the need for role separation or external validation
  • that the architecture is portable across models and tool surfaces
  • that transcript export is a core continuity requirement
  • that the system must be run by a human SME with software-engineering and DR instincts
  • that external systems of record remain essential validators

Why The Human Role Must Stay Central

These memos continue to assume a hard truth already present in the earlier set: the architecture does not work because a model is smart in the abstract. It works because a human subject-matter expert with systems discipline is using specialized agent lanes in a controlled way.

The needed skill stack is still unusually broad:

  • subject-matter expertise
  • software engineering judgment
  • backup and disaster-recovery thinking
  • workflow decomposition into specialized roles
  • taste, acceptance criteria, and willingness to reject model output

Durable Takeaway

The March memos described the transition from exploratory AI assistance into a structured local operating system. These May memos describe the next stage: the same architecture after it proved capable of publication, cross-model transfer, and survival-oriented thinking in a hostile vendor environment.

The architecture should now be understood not just as a productive AI workflow, but as a recoverable multi-lane local system whose memory must remain under local human control and whose outputs can survive contact with the real world in the form of working public artifacts.
