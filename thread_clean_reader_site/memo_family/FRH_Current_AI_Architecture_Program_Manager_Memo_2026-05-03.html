FRH Current AI Architecture - Program Manager Memo

Date: 2026-05-03

Purpose

This memo explains the current FRH AI architecture in program-manager terms after the Codex FRH agent experiment, the addition of Web_Reader, and the discovery that transcript export is a continuity requirement rather than an optional convenience.

Executive Summary

The FRH project still uses the same core role-separated architecture described in the March memo set, but it is now stronger in three important ways:

  • it has a real publication lane
  • it has proven portable across multiple model and tool environments
  • it now treats full local export of agent chat transcripts as a disaster-recovery control

The key lesson remains the same: AI becomes useful in serious work when it is wrapped in an operating model. What changed is that the operating model has now proven itself under more realistic pressure, including cross-tool movement, publication workflows, and explicit vendor-risk concerns.

The Current Lane Model

The architecture now has five visible functional lanes:

  • CTC - primary operator/build lane
  • GMA - higher-order interpretive and scenario review lane
  • Parser - safe structural adapter to legacy truth formats
  • DR_Lab - backup, continuity, rollback, and recovery lane
  • Web_Reader - staging, publication, and presentation lane

That is an important evolution. Earlier memos described a productive local architecture. The current state describes a productive local architecture with a completed downstream publishing path.

What It Has Already Delivered

This architecture has now produced two strong real-world outputs that make the value proposition easier to explain:

  • a Sample Character Library website flow built from locally engineered character artifacts, including images, HTML, and refreshed PDFs, then promoted to Web_Reader and published to GitHub Pages
  • a New Player Packet website flow built from explanatory and onboarding material, reauthored into a web-readable structure locally and then polished and published through Web_Reader

That matters because these are not toy outputs. They are high-quality, public-facing deliverables emerging from a knowledge domain that is unusually computer-unfriendly: a custom tabletop role-playing framework built on subtle setting logic, narrative design, and legacy mechanical artifacts.

In program-manager terms, this means the architecture has crossed a crucial threshold. It is not merely helping an expert think faster. It is coordinating multiple specialized lanes to produce publishable assets.

Why This Matters Operationally

Many AI writeups fail because they stop at the idea-generation layer. FRH no longer stops there. It now demonstrates a full working chain:

authoring - review - structural handling - validation - recovery - publication

That makes the architecture easier to explain to technical leadership, operations staff, and management because it is no longer just a creative workflow. It is a small but real operating system around difficult expert work.

What The Codex Experiment Proved

The Codex-centered CTC lane showed that direct local workspace access, patching, shell usage, and artifact-aware execution improve reliability and throughput. However, the experiment did not eliminate the earlier architectural rules. It reinforced them.

The same controls are still required:

  • human SME oversight
  • external systems of record
  • role separation
  • durable artifacts
  • backup and recovery discipline

In program-manager language, the tool got better, but the operating model was what made the tool safe to trust.

The Most Important New Discovery

The single most important cross-model and DR discovery is the need to export full agent chat transcripts to local files.

This must now be treated as a standard control because:

  • vendors can raise token costs unexpectedly
  • vendors can close or restrict accounts without warning or explanation
  • project continuity must not depend on a hosted conversation history remaining accessible
  • cross-model transfer requires locally owned operational memory

For management, this is the key continuity point: if the architecture's memory lives only inside a vendor account, then the architecture does not truly belong to the operator.

Portability Is Now Proven

The FRH architecture has now shown meaningful portability across:

  • Codex
  • Claude Code
  • Gemini
  • and even limited local-model experiments such as Gemma 4 E4B on consumer hardware

This matters because it shifts the value proposition away from any single vendor. The real durable asset is the workflow architecture and its local artifact base, not a particular model subscription.

What Skill Stack Is Required

These results do not come from generic prompt usage alone. The architecture still depends on an unusually broad human capability stack:

  • deep subject-matter expertise
  • software and workflow engineering
  • IT backup and disaster-recovery instincts
  • ability to decompose the work into specialized agent lanes
  • willingness to validate against external truth instead of trusting the model's confidence

This is a useful management lesson by itself. The system is not self-running. It is a human-led expert system with AI helpers.

Business Translation

The business pattern is still recognizable:

  • keep the trusted local system of record
  • wrap it with safe adapters and specialized helpers
  • preserve durable local artifacts
  • export the operational memory so continuity does not depend on one vendor
  • maintain recovery-ready backups and deltas

A concrete example is a small business stuck on an older local Peachtree Accounting installation that is too risky or too expensive to replace immediately.

The FRH lane model maps cleanly:

  • Parser becomes the safe interface layer for exports, imports, structured edits, and validation around the accounting system's files and reports
  • CTC becomes the day-to-day operator helper for invoices, bookkeeping support, customer correspondence, procedural reminders, and routine documentation
  • GMA becomes the higher-order reviewer for anomalies, workflow sanity, trends, and management-facing interpretation
  • DR_Lab becomes the continuity layer for backups, restores, rollback planning, and verification of recovery readiness
  • Web_Reader can become the publishing or presentation lane for client-facing statements, internal portals, reporting dashboards, or cleaned management summaries

The point is not that AI replaces the accounting system. The point is that AI can wrap the legacy system with a more legible, more recoverable operating model while the authoritative business records stay where they already are.

This remains highly portable to small-business and legacy-software environments where replacement is risky but productivity pressure is real.

Closing Summary

The current FRH architecture should be understood as a recoverable, role-specialized, locally anchored AI operating model with a publication lane and proven cross-tool portability.

The most important management-facing lessons are:

  • AI needs a disciplined operating system around it
  • the human SME remains central
  • external validation is mandatory
  • transcript export is now part of disaster recovery
  • the architecture can produce real public outputs even in difficult domains
  • and its real value lies in portability and recoverability, not in dependence on one model vendor
