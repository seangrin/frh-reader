FRH Current AI Architecture - Technical Overview

Date: 2026-05-03

Abstract

This document describes the current FRH AI architecture after the Codex FRH agent experiment and after the addition of Web_Reader as a first-class publication lane. The architecture remains a role-specialized local operating system built around persistent artifacts, external validators, and recovery discipline. The major technical update is that cross-model portability and transcript-export-based continuity have now become core architectural concerns.

1. Purpose

The purpose of this architecture is to make AI assistance useful in a difficult, high-context local workflow without allowing the project to collapse into ephemeral vendor chat, silent drift, unverifiable output, or unrecoverable operational state.

The architecture exists to improve:

  • throughput
  • durability
  • recoverability
  • publication readiness
  • and cross-model survivability

2. Current Problem Shape

FRH remains a difficult AI support problem because it combines:

  • niche creative and analytical domain work
  • high-context long-lived project memory
  • legacy local systems of record
  • file-heavy iteration
  • human taste and acceptance thresholds
  • need for recoverable handoffs
  • and now downstream website publication

The architecture therefore cannot be reduced to one model prompt or one browser session. It is a multi-lane local system with persistent state outside the model.

3. High-Level Architecture

The current FRH system consists of:

  • one human operator who is the subject-matter expert and final authority
  • multiple specialized agent lanes
  • a durable local artifact tree
  • external validators and systems of record
  • backup, delta, and checkpoint infrastructure
  • a publishing lane for web-facing delivery
  • cross-model portability through exported local memory

The simple conceptual loop is:

human intent - specialized lane work - external validation - local persistence - recovery capture - publication staging

4. Demonstrated Output Classes

The architecture should now be evaluated partly by what it has already produced.

The strongest demonstrated output classes so far are:

  • character-engineering and publication bundles, where valid character artifacts, reader-facing HTML, images, and PDFs were coordinated locally and then promoted for web publication
  • player-facing onboarding websites, where dense and irregular source material was transformed into coherent web-facing explanatory pages

Two concrete proof cases are especially important:

  • the Sample Character Library, primarily coordinated through CTC and then handed to Web_Reader for polishing and GitHub Pages publication
  • the New Player Packet website, built from FRH onboarding material with GMA-adjacent interpretive help and CTC authoring, then finalized through Web_Reader

This matters technically because it shows the architecture can convert a highly irregular and concept-heavy local corpus into consistent public-facing deliverables without losing recoverability.

6. Artifact And Memory Model

The architecture still relies on local artifacts for operational memory. These include:

  • source notes
  • batch logs
  • handoff memos
  • source maps
  • prompt records
  • staging folders
  • HTML outputs
  • checkpoint manifests
  • session status notes

The architecture's memory is intentionally externalized into files. This remains one of the main reasons the system is portable and recoverable.

7. Source Of Truth And Authority Model

The authority model remains strict:

  • the live local workspace trees are the actual working source of truth
  • legacy systems of record remain authoritative for the domains they own
  • published outputs are not assumed to outrank source work folders
  • mirrors and backup chains exist for recovery, not to create authority ambiguity

The most important technical discipline is still: the model does not get to declare truth by confidence alone.

8. Validation Model

The AI lanes are not their own validators. External systems of record remain mandatory. In the FRH workflow, this includes legacy backend tools, local source artifacts, and other durable non-model authorities.

This validation rule matters even more in a cross-model world. When the same workflow can move between Codex, Claude Code, Gemini, and local-model experiments, the validation authority must remain outside the model family entirely.

9. Publication Model

Web_Reader created a distinct downstream pattern:

work folder - verification - promotion - web staging - publication

This means the architecture now separates:

  • authoring truth
  • review truth
  • structural mutation truth
  • recovery truth
  • presentation truth

That separation helps prevent web polish from contaminating the authoring pipeline and helps preserve explainable handoffs.

10. Recovery And Continuity Model

The architecture remains strongly recovery-oriented. This includes:

  • checkpoint and delta capture
  • backup chains
  • promotion discipline
  • session-state notes
  • handoff documents
  • non-destructive iteration habits

However, the recovery model is now broader than file backup alone. It explicitly includes preservation of the conversational and procedural memory that shaped the artifacts.

11. Transcript Export As DR Control

This is the most important new technical rule since the March memos.

Full agent chat transcript export to local files is now part of the continuity model because:

  • hosted conversation history is not a reliable long-term asset
  • vendors may change pricing abruptly
  • vendors may close or restrict accounts without warning
  • future continuation may require another model or tool environment

Therefore the architecture must preserve not only outputs, but also the operative reasoning trail, task history, and method evolution in locally owned artifacts.

Stated bluntly: if the transcript is not locally exported, then the project's live procedural memory is still vendor-hosted and can be lost.

12. Portability Across Models And Tools

The architecture has now shown practical portability across:

  • Codex and the Codex desktop tool surface
  • Claude Code and its mirrored FRH workspace family
  • Gemini-assisted lanes
  • limited local-model experimentation, including Gemma 4 E4B with 8K context on a 3080

This implies that the durable part of the system is:

  • the workflow decomposition
  • the artifact tree
  • the validation model
  • the recovery model

Model choice remains important, but it is now clearly subordinate to the architecture.

13. Human Skill Stack Requirements

The architecture still depends on a human operator with a broad combined skill stack:

  • subject-matter expertise
  • software engineering
  • artifact and workflow design
  • backup and disaster-recovery practice
  • ability to divide knowledge domains into specialized agent lanes
  • taste and acceptance judgment

This requirement must remain explicit because the architecture can otherwise be misunderstood as a generic model trick. It is not. It is an expert-run system.

14. Current Strengths

The architecture is now strong at:

  • multi-lane local work
  • artifact-driven memory
  • publication staging
  • cross-model portability
  • checkpoint-aware operations
  • recoverable handoffs
  • preserving domain truth under tool changes
  • producing high-quality public deliverables from a difficult custom knowledge domain

15. Current Risks

The biggest risks are no longer just parser gaps or workflow roughness. They also include platform and vendor risk:

  • pricing instability
  • account closure or lockout
  • loss of hosted conversation memory
  • confusion between publication copies and work-folder truth
  • insufficient transcript export discipline

16. Why This Architecture Still Works

It works for the same reasons the March version worked:

  • the human expert remains central
  • specialized lanes reduce role confusion
  • local artifacts hold memory
  • external validators remain authoritative
  • recovery is built in rather than assumed later

What changed is that publication and transcript preservation are now clearly part of that same design logic.

17. Transferability

This pattern remains portable to business environments built around legacy local systems. The stronger current claim is that portability is not only theoretical. The FRH project has already shown tool-surface migration and partial model migration while preserving the same architecture.

A concrete business analogy remains useful: a small business still running on local Peachtree Accounting.

In that mapping:

  • Parser acts as the safe adapter around the local accounting truth, handling exports, structured reads, checked transformations, and boundaries on write-back behavior
  • CTC acts as the execution lane for day-to-day office work around the system, such as invoice support, customer communication drafts, process notes, and operator assistance
  • GMA acts as the analytical and review lane, reading outputs for trends, anomalies, and higher-order business interpretation
  • DR_Lab acts as the backup, restore, rollback, and survivability lane
  • Web_Reader acts as the downstream presentation lane for management reports, customer-ready summaries, or internal web-facing views built from validated local outputs

That is why this architecture is portable: it does not depend on replacing the old system. It depends on preserving the old system as the record of truth while surrounding it with recoverable specialized lanes.

18. Current Operating Summary

The current FRH architecture is best understood as a local, role-specialized, artifact-driven, human-led operating system for expert work, with:

  • CTC, GMA, Parser, DR_Lab, and Web_Reader as distinct lanes
  • external systems of record and validators
  • checkpoint and delta continuity
  • portable workflow design across multiple model vendors and tool surfaces
  • and mandatory export of agent chat transcripts to local files as part of disaster recovery

The system should no longer be described merely as AI assistance. It is now a recoverable multi-agent local architecture that can produce publishable public artifacts from a difficult custom domain while keeping its memory and methods under local human control even if vendors change.
